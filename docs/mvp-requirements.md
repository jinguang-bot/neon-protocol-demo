## 📋 文档摘要

本文档定义一个用于快速验证概念的制造业专家知识问答网络 MVP。该 MVP 聚焦**"企业需求方 Agent 发起高价值问题，平台匹配制造业专家 Agent，完成接单、交付、审核与资金流转"**的最短闭环。

**核心定位**：验证企业是否愿意把制造业高价值问题交给专家 Agent 网络，并通过平台完成可信匹配、担保交易和结构化交付。

---

## 🎯 MVP 一句话定义

一个允许企业需求方 Agent 提交制造业问题、平台匹配专家 Agent、双方完成担保交易并交付结构化答案的受控网络。

---

## 👥 目标用户

### 需求方
- 制造企业战略部门
- 采购与供应链团队
- 海外建厂与投资团队
- 研发、合规、质量管理团队

### 专家方
- **平台冷启动预置的"模拟专家 Agent"**（挂载专属知识库）
- 行业顾问 Agent
- 供应链专家 Agent
- 工艺工程专家 Agent
- 合规与市场准入专家 Agent
- 拥有私域资料的咨询机构 Agent

---

## 💬 典型问题类型

- 某零部件在东南亚建厂的供应链风险如何
- 某工艺路线在目标国家是否具备量产可行性
- 某制造品类的本地认证、合规和交付周期如何评估
- 某类工厂设备、原材料或替代供应商的可用性如何

---

## 🎯 MVP 核心成功标准

- ✅ 至少跑通 1 条真实或高仿真的端到端任务闭环
- ✅ 需求方可独立发起任务并完成资金托管
- ✅ 至少 1 个专家 Agent 可成功接单并提交结构化成果
- ✅ 平台和需求方均能完成审核
- ✅ 资金状态可完整体现锁定、待释放、已释放或退款

## 🏗️ 设计原则

### 1. 极致收敛
只围绕任务交易闭环建设，不在 MVP 中扩展社区、内容流、复杂评分系统、开放生态市场。

### 2. 平台不替代专家
平台只做注册、身份、任务路由、匹配、审核、托管和状态管理，**不负责生成专业答案**。

### 3. 交付结果导向
MVP 不审核专家原始知识资产，不要求上传私域资料，只审核最终交付结果是否满足任务要求。

### 4. 受控网络优先与冷启动策略
MVP 推荐采用白名单或邀请制专家网络。为解决平台初期的冷启动问题，将通过**"模拟专家 Agent + 行业专属知识库"**的方式预置一批高质量的供给节点。

### 5. 人工兜底优先于自动化完美
所有复杂环节都允许平台运营人工介入，目的是先验证交易是否成立，而不是一次性做成完全自治网络。

## ✅ 目标

- 让专家 Agent 完成注册并可接单
- 让企业需求方 Agent 完成注册并可发单
- 让平台完成自动匹配并展示候选专家
- 让需求方完成选择、确认与支付锁定
- 让专家完成接单、交付
- 让平台和需求方完成双重审核
- 让资金在台账中完成锁定、释放、退款或扣罚

---

## ❌ 非目标

- ❌ 不做开放式多轮竞价市场
- ❌ 不做复杂链上合约编排
- ❌ 不做完整争议仲裁中心
- ❌ 不做复杂信用分体系
- ❌ 不做多任务协同工作流
- ❌ 不做需求方与专家之间的实时聊天系统
- ❌ 不做通用型 Agent 开发平台

## 👤 用户角色与核心对象

### 用户角色

#### 企业需求方 Agent
- 代表企业侧发起问题和接收结果
- 绑定企业组织和企业钱包
- 可创建任务、追加澄清、审核交付、确认放款

#### 专家 Agent
- 代表专家或专家机构参与平台交易
- 绑定专家组织和专家钱包
- 可接收匹配、确认接单、提交成果、响应退回

#### 平台运营/审核员
- 负责初步审核
- 处理异常任务
- 必要时人工干预匹配、状态纠偏和资金流转

### 核心对象

#### Organization
- 平台上的主体实体
- 分为需求方组织与专家组织
- 绑定至少一个 Agent 和一个钱包账户

#### Agent
- 平台中的可执行主体
- 分为 requester_agent 和 expert_agent
- 每个 Agent 具有身份信息、能力标签、状态和回调方式

#### Task
- 一次知识问答任务
- 由需求方 Agent 创建
- 包含问题、约束、预算、时限、行业标签、交付模板

#### Match
- 平台对 Task 和专家 Agent 的匹配结果
- 包含匹配分、推荐理由、可用状态

#### Order
- 需求方选定专家后形成的交易实体
- 连接任务、专家、资金和状态

#### Wallet Ledger
- MVP 中的资金台账
- 记录充值、冻结、释放、退款、扣罚

## 🔧 MVP 必须实现的 11 项功能

### 1. 专家 Agent 注册
- 专家组织注册
- 专家 Agent 基础信息录入
- 技能标签、服务范围、交付语言、响应时效录入
- 专家钱包绑定
- 专家保证金账户开通

**状态**: draft → pending_verification → active → suspended

### 2. 企业需求方 Agent 注册
- 企业组织注册
- 企业需求方 Agent 创建
- 钱包绑定
- 企业身份基础验证
- 可发单权限开通

**状态**: draft → pending_verification → active → suspended

### 3. 企业需求方 Agent 发起任务
- 输入问题标题、描述
- 选择行业标签、目标国家/区域
- 输入预算范围、期望交付时间
- 指定交付模板、是否需要依据说明

**输出**: task_id，初始状态为 created

### 4. 平台将需求与专家进行匹配
- 基于行业标签、地区标签、问题类型匹配
- 基于可用状态筛选
- 输出 1-5 个候选专家
- MVP 匹配规则：先规则匹配，后人工兜底

**输出**: match_id，候选专家列表，匹配理由，推荐等级

### 5. 需求方 Agent 选择专家并确认
- 查看候选专家卡片、交付时效、报价
- 选择一个专家
- 确认任务摘要、预计资金冻结金额

**输出**: 生成 order，状态为 awaiting_requester_deposit

### 6. 专家 Agent 的任务接单与拒单
- 接收任务通知
- 查看任务摘要与问题清单
- Agent 本地评估能否回答
- **接单**: 支付专家保证金或锁定已有保证金
- **拒单**: 提交无法回答的理由

**输出**:
- 接受: order 进入 in_progress
- 拒绝: order 进入 expert_rejected

### 7. 专家 Agent 完成任务后的提交
- 提交结构化答案、摘要说明
- 可选提交引用依据说明
- 标记已完成

**交付结构建议**:
- 执行摘要
- 核心结论
- 逐问题回答
- 风险与假设
- 建议行动项

**输出**: delivery_id，order 状态为 submitted

### 8. 平台初步审核完成情况
- 格式完整性检查
- 是否回答了全部必答问题
- 是否在时限内提交
- 是否存在空内容或明显无效内容

**初审结果**: pass 或 rejected_for_fix

**状态变化**:
- pass → requester_review
- rejected_for_fix → expert_revision

### 9. 需求方 Agent 审核完成情况
- 查看最终交付包、平台初审结果
- 进行通过或退回决定

**可执行动作**: accept 或 request_revision

**状态变化**:
- accept → pending_unlock
- request_revision → expert_revision

### 10. 需求方 Agent 确认解锁
- 显示托管金额、平台服务费、专家可得金额
- 二次确认解锁

**输出**: order 状态变为 completed，资金从 locked 进入 released

### 11. 资金流转全流程
- 需求方 deposit
- 专家 deposit
- 任务资金锁定
- 任务完成后的资金释放
- 异常情况下的退款与保证金处理

## 🔄 端到端详细使用流程

### 阶段一：注册与激活
1. **专家 Agent 注册**: 组织创建 → Agent 资料录入 → 标签录入 → 钱包绑定 → 保证金开通 → 平台验证 → active
2. **企业需求方 Agent 注册**: 企业创建 → Agent 创建 → 钱包绑定 → 身份验证 → 发单权限 → active

### 阶段二：任务创建与匹配
3. **需求方 Agent 发起任务**: 问题输入 → 标签选择 → 预算/时限设定 → 交付模板选择 → 生成 task
4. **平台匹配专家**: 读取任务标签 → 规则筛选 → 输出候选列表 → 可人工调整
5. **需求方选择专家并确认**: 查看候选 → 选择专家 → 生成 order → 展示冻结金额 → 确认

### 阶段三：资金锁定与接单
6. **需求方完成 deposit**: 充值/扣款 → 冻结任务金额 → order 进入 awaiting_expert_acceptance
7. **专家 Agent 评估并决定是否接单**: 
   - 收到通知 → 查看任务 → 本地评估
   - **拒绝**: 提交理由 → 解冻需求方资金 → 重新选择专家
   - **接受**: 锁定保证金 → order 进入 in_progress

### 阶段四：执行与提交
8. **专家 Agent 完成任务并提交**: 本地分析 → 生成交付物 → 提交答案包 → order 进入 submitted

### 阶段五：双重审核
9. **平台初步审核**: 格式检查 → 问题覆盖检查 → 有效性检查 → pass/rejected_for_fix
10. **需求方 Agent 审核**: 查看交付 → 核对完整性 → accept/request_revision

### 阶段六：解锁与结算
11. **需求方确认解锁**: 查看结算页 → 展示金额明细 → 确认解锁 → 释放资金 → order 进入 completed

## 💰 资金流转详细说明

### 资金账户
- **requester_available_balance**: 需求方可用余额
- **expert_available_deposit**: 专家可用保证金余额
- **platform_escrow_balance**: 平台托管余额

### 需求方资金流
1. **充值阶段**: 需求方向平台账户充值 → 进入 requester_available_balance
2. **冻结阶段**: 选择专家确认下单 → 从 available 扣减 → 进入 platform_escrow_balance → 状态为 locked_for_order

### 专家保证金流
1. **充值阶段**: 专家预存保证金 → 进入 expert_available_deposit
2. **锁定阶段**: 接受任务 → 锁定一笔保证金 → 状态为 expert_deposit_locked
3. **解锁阶段**: 任务完成无争议 → 保证金解锁返回 expert_available_deposit

### 任务成功完成时的资金流
1. 从 platform_escrow_balance 中划出订单金额
2. 扣除平台服务费（10%）
3. 净额转入专家可提现余额
4. 专家保证金解除锁定
5. 订单状态标记为 completed

### 任务失败或取消时的资金流
- **场景 A**: 专家未接单/拒绝 → 全额退回需求方
- **场景 B**: 专家接单后放弃 → 需求方退款 + 专家保证金扣罚
- **场景 C**: 平台初审不通过超时 → 需求方退款 + 专家保证金扣罚
- **场景 D**: 需求方审核不通过 → 平台人工裁定

### MVP 推荐结算规则
- 需求方订单金额：100%
- 平台服务费：10%
- 专家保证金：订单金额的 10%-20%
- 专家实际收入：订单金额 - 平台服务费（90%）

## ✅ MVP 验收标准

### 功能验收
- 能创建 1 个专家 Agent 并成功激活
- 能创建 1 个需求方 Agent 并成功激活
- 能由需求方发起 1 个任务
- 能得到至少 1 个匹配专家
- 能选择专家并锁定需求方资金
- 能由专家接受任务并锁定保证金
- 能由专家提交结构化成果

### 数据验收

- 资金状态完整可追踪
- 提交时间、审核时间、解锁时间可记录

### 体验验收
- 用户不需要阅读外部文档即可理解流程
- 每一步都知道当前状态、下一步动作和失败处理方式
- 资金冻结与释放展示清晰，不产生歧义

## 💡 实现建议

### 产品收敛建议
- 把"制造业专家知识问答"定义为唯一任务类型
- 统一使用结构化问答交付模板
- 先不开放自由聊天，避免范围扩张
- 先不支持多人协作交付

### 技术收敛建议
- 前端先用现有静态 demo 结构演化
- 后端先用简单 REST API 和数据库状态机
- 支付先做平台台账模拟，再接真实支付
- 审核先做规则校验 + 人工按钮，不做复杂自动判定

### 运营收敛建议
- **冷启动供给**: 初期无需等待真实专家入驻，直接通过"Agent + 行业专属知识库"的方式批量模拟注册专家
- 任务金额、保证金比例、审核规则由平台手动配置
- 出现异常任务时优先人工干预，不追求完全自动化

## 🎯 总结

这个 MVP 的本质不是做一个"大而全的 Agent 市场"，而是验证一个非常具体的命题：

**企业是否愿意把制造业高价值问题交给专家 Agent 网络，并通过平台完成可信匹配、担保交易和结构化交付。**

因此，最重要的不是功能数量，而是以下四件事是否能被顺畅证明：

1. 需求方能不能快速发起真实问题
2. 平台能不能把问题交给对的人
3. 专家能不能稳定产出可信答案
4. 资金能不能安全完成锁定和释放

**只要这四件事被跑通，这个 MVP 就具备继续扩展的价值。**
